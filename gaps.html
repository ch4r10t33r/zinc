ZINC website — gap analysis

Scope: the whole site except blog posts — homepage, /zinc/ landing, /about, the docs hub + all docs/*.md pages, the Zig API reference, and the benchmarks/performance dashboards. Generated 2026-05-24. Analysis only — no site changes were made. Numbers cited were re-verified against site/src/data/zinc-performance.json and the source files on this date.

Bottom line. The site is genuinely good — honest tone, real benchmarks, deep docs, clean visuals. The gaps are not "it's broken"; they're "a motivated newcomer or developer hits avoidable friction, and a skeptic finds things that quietly undercut trust." The four highest-leverage themes:
  1. Conversion safety-rails for first run — Windows expectation, the RADV_PERFTEST requirement, and a single copy-paste quickstart are all missing from the critical path.
  2. The Intel story is incoherent — the landing page says "AMD + Apple," yet Intel is published in the docs and in the benchmark dashboard, where 4 of 5 models are failing and the data is 5 days stale.
  3. "Shipped vs. aspirational" is unlabeled — design/planning docs sit alphabetically beside user guides; ZINC_RT, TurboQuant, and continuous batching read as available in some places and roadmap in others.
  4. Duplicate benchmark page + thin trust scaffolding — /zinc/performance is a canonical-duplicate of /zinc/benchmarks, and methodology caveats (differing llama.cpp builds, failed rows) are a click or two too deep for a skeptic.

Executive summary — the blockers

If only a handful of things get fixed, these are the ones that either stop a real person from succeeding or actively erode trust:

  1. Windows visitors aren't told early they're out of scope (engine is Linux/AMD or macOS/Apple only). They invest, then fail.
  2. RDNA4 first-run quietly fails without RADV_PERFTEST=coop_matrix, and that requirement isn't in the main quickstart command block.
  3. No troubleshooting landing spot — when the first run fails, there's no "here's what to check" section near where the user is.
  4. Intel target is 80% broken and 5 days stale, but still presented in the dashboard next to an "AMD + Apple" marketing message — a skeptic reads this as silent failure / cherry-picking. verified: 4/5 Intel ZINC runs failed (execbuf ioctl ENOMEM), intel gen 2026-05-19 vs rdna 2026-05-24
  5. API integrators hit two unstated walls: no /v1/embeddings, and decode is serialized (concurrent requests queue) — both true, both buried.
  6. /zinc/performance is a duplicate of /zinc/benchmarks that declares the other as canonical — split link-equity and a confusing second URL. verified in performance.astro

A. New-user journey

Persona: someone who heard "fast local LLM engine for AMD GPUs," lands on the site, and tries to go from zero → running a model → chatting.

Blocker A1 · Windows isn't ruled out early hurts: Windows visitors (large segment)

The landing hero and tagline say "AMD GPUs & Apple Silicon" with no OS qualifier. The Linux-only requirement for AMD is buried mid-sentence in Hardware Requirements; Getting Started's "before you start" list has no explicit "Windows: not supported" line.

Evidence: site/src/pages/zinc/index.astro hero/tagline; docs/HARDWARE_REQUIREMENTS.md:3; docs/GETTING_STARTED.md:15–20.

Fix: one bold line near the top of /zinc/ and Getting Started: "Runs on Linux (AMD/Intel GPUs) or macOS (Apple Silicon). Windows is not supported."

Blocker A2 · RADV_PERFTEST=coop_matrix missing from the first-run command hurts: every RDNA4 Linux first-timer

Getting Started is a linear narrative, but the first actual inference command doesn't include the env var; the requirement appears only in the preflight section and in a paragraph after the example. Without it, RDNA4 runs can fail or misbehave — the user blames ZINC.

Evidence: command at docs/GETTING_STARTED.md:106; requirement at :74–79 and :111–116; also docs/RUNNING_ZINC.md:28–31.

Fix: put export RADV_PERFTEST=coop_matrix directly in the main quickstart block, bolded, before the first run.

Blocker A3 · No troubleshooting path where the user actually is hurts: anyone with a non-stock setup

"Common early mistakes" and "fast troubleshooting commands" exist only at the end of Running ZINC, with no actionable next steps and no link to Discord/issues. A cryptic Vulkan/driver/VRAM error leaves the user with nowhere to go.

Evidence: docs/RUNNING_ZINC.md:445–462 (mistakes, no remedies), :464–479 (commands, buried). No top-of-page Troubleshooting section anywhere.

Fix: a dedicated "Troubleshooting" section early in Running ZINC, per failure mode (wrong driver, unsupported HW, missing env var, model too large), each with a diagnostic command and a help link.

Friction A4 · No single copy-paste "zero → chat" recipe hurts: impatient newcomers

All the steps exist (clone → build → check → pull → run → chat) but are spread across sections with no numbered "do these in order" block. A newcomer can't skim-and-go.

Evidence: docs/GETTING_STARTED.md overall structure; steps not consolidated.

Fix: a "Quick start (5 minutes)" fenced block at the very top with ~8 commands in order.

Friction A5 · Model acquisition is fragmented & "what is a managed model?" is unclear hurts: first-time model load

zinc model pull qwen35-9b-q4k-m is shown without explaining what the catalog is, where files come from, or that arbitrary -m /path/model.gguf GGUFs also work. The managed-vs-manual distinction and HuggingFace links live in three different places.

Evidence: docs/GETTING_STARTED.md:26–32 (table w/ HF links), :93–99 (pull), docs/RUNNING_ZINC.md:133–147,172,195–198.

Fix: one "Finding & loading models" section: what the catalog is, where it downloads from + SHA-256 verification, and the bring-your-own-GGUF path.

Friction A6 · "Will it run on MY card?" has no crisp answer hurts: hardware-decision visitors

VRAM guidance is qualitative ("comfortably," "might be tight," "fits on 24+ GB") with no rule of thumb, no context-length trade-off, and no clear pointer that zinc --check -m <model> prints an actual fit estimate.

Evidence: docs/HARDWARE_REQUIREMENTS.md:46–51,116–124; docs/RUNNING_ZINC.md:51 (mentions --check fit but doesn't teach reading it).

Fix: state the --check -m workflow prominently; add a short "VRAM vs model vs context" rule of thumb.

Friction A7 · "Experimental" warning is inconsistent & absent from the homepages hurts: expectation-setting

The experimental caveat is a blockquote in Getting Started and a mid-page bold line on /zinc/, but there's nothing on the site root / and no visually prominent callout. Easy to read "fast local inference" as production-ready.

Evidence: docs/GETTING_STARTED.md:1–3; site/src/pages/zinc/index.astro mid-body; site/src/pages/index.astro (none).

Fix: a consistent callout component on /zinc/ (and a line on /).

Polish A8 · --chat flag vs. chat subcommand can confuse hurts: clarity

--chat (wrap prompt in chat template) and chat (start the web UI server) appear close together with no note that they're different things.

Evidence: docs/GETTING_STARTED.md:103–109 vs :129–143.

Fix: one clarifying sentence at first mention.

Polish A9 · No platform-choice guidance (AMD vs Apple) hurts: undecided buyers

The site explains why both platforms are targeted but never helps a reader choose. The data actually tells a clear story (RDNA4 flagship 128.5 tok/s and beating llama.cpp; Metal flagship 33.9 and trailing) but it's never summarized as "pick AMD for speed, Apple if you already own a Mac."

Evidence: site/src/pages/zinc/index.astro intro + per-platform boards; docs/HARDWARE_REQUIREMENTS.md (separate sections, no comparison).

Fix: a 2-row "choosing a platform" comparison.

Polish A10 · Discord help link absent from docs & /about hurts: stuck users

The Discord invite lives on /zinc/ only; doc pages and /about don't surface any community/help link, so a stuck reader has to navigate back.

Evidence: site/src/pages/zinc/index.astro (has it); docs + site/src/pages/about.astro (don't).

Fix: add a "Getting help" link in the doc layout footer.

B. Developer & API-integrator journey

Persona (1): integrator pointing an OpenAI client at ZINC's server. Persona (2): contributor building from source & navigating the engine.

Blocker B1 · /v1/embeddings is unsupported — but silently so hurts: RAG/search integrators

The endpoint table omits embeddings; the disavowal is a single line near the end of the API doc. An integrator wires up a RAG pipeline and discovers the gap mid-build.

Evidence: docs/API.md:35 (table omits it), :494 (buried "does not expose"); docs/SPEC.md:565–575.

Fix: a "Not supported yet" box near the top of the API doc (embeddings, plus any other OpenAI gaps).

Blocker B2 · Serialized decode isn't stated where integrators look hurts: anyone sending concurrent requests

Concurrent HTTP is accepted but decode runs behind one lock. The only clear statement is one line in API "compatibility notes"; the reason lives 20 pages deep in SPEC. An integrator prototypes 4 parallel requests, sees them queue, and assumes a bug.

Evidence: docs/API.md:29–30; docs/SPEC.md:496–508.

Fix: a "Concurrency & throughput" note up top in the API doc; reference continuous-batching roadmap status.

Blocker B3 · ZINC_RT reads as a shippable backend without the caveat hurts: contributors picking a target

-Dbackend=zinc_rt is a real build option and ZINC_RT_DESIGN is extensive ("M0 shipped"), but the headline 80-tok/s figure is a host-assisted, clamped path (caps LM-head rows, clamps MoE top-k after prefill). README/landing don't mention ZINC_RT at all, so the relationship to the production Vulkan path is unclear.

Evidence: docs/ZINC_RT_DESIGN.md:1–10,85–91; build.zig backend option; README has no ZINC_RT mention.

Fix: one "status: not the default hot path; Vulkan is" banner at the top of the design doc, and a one-liner wherever the backend flag is documented.

Friction B4 · No real OpenAI-SDK example (Python/JS) hurts: integrators

The API doc shows only curl/JSON. Integrators must guess base-URL/key setup for the actual SDKs the page claims to support.

Evidence: docs/API.md:1–4,214–233.

Fix: add a Python (openai) and JS snippet with base_url + dummy key.

Friction B5 · No "architecture in 10 minutes" narrative hurts: new contributors

SPEC is long; DEVELOPMENT lists an ASCII tree; the Zig API page is a symbol browser. There's no short call-stack story (CLI → GGUF load → forward pass → GPU dispatch → sampling → HTTP routes), so a contributor reads source for hours to orient.

Evidence: docs/DEVELOPMENT.md:90–159; docs/SPEC.md:43–55; no ARCHITECTURE.md.

Fix: a one-page code-flow walkthrough linking into the Zig API entry modules.

Friction B6 · Build toolchain versions are partly vague; no tested matrix hurts: from-source builders

Zig is pinned ("0.15.2+") but Bun is "latest" and glslc/shaderc guidance is loose, with no statement of which versions CI actually tests. A shader-compiler regression looks like "works locally, fails in CI."

Evidence: docs/DEVELOPMENT.md:7–16; README prereqs; no version guard in build.zig.

Fix: publish a tested-versions table (Zig/Bun/glslc/Vulkan) and note known-bad combos.

Friction B7 · Contributing/License/issues not linked from the site hurts: would-be contributors

CONTRIBUTING.md / CODE_OF_CONDUCT.md exist in-repo but nothing on the docs site (incl. the Zig API "entry points" page) points to PR process, license, or the issue tracker.

Evidence: site/src/pages/zinc/docs/zig-api/index.astro; docs/DEVELOPMENT.md:282–286 (repo-relative links only).

Fix: a "Contributing" link in the docs nav/footer pointing to the GitHub docs.

Hygiene B8 · Zig API entry points are buried below the fold hurts: engine hackers

The reference is rich and the curated entry modules (forward, graph, gguf, routes, …) are correct — but they sit beneath stat cards, search UI, and a supplemental-docs grid, so a developer lands in a 100-symbol browser instead of "start here."

Evidence: site/src/pages/zinc/docs/zig-api/index.astro:54,154–172.

Fix: hoist "Developer entry points" above the stats/search.

Hygiene B9 · Test scope & GPU-profiling workflow undocumented hurts: contributors validating changes

Test commands are listed but not what they cover (fast vs. slow, which env var gates which suite via ZINC_REQUIRE_FULL_TESTS). Profiling flags (ZINC_PREFILL_PROFILE, graph export, tools/render_graph_report.ts) are named but there's no "measure a kernel regression" walkthrough.

Evidence: docs/DEVELOPMENT.md:49–90; build.zig test wiring; docs/SPEC.md:221–256.

Fix: document the test tiers and a short profiling recipe.

Hygiene B10 · Roadmap vs. SPEC vs. README disagree on feature phases hurts: contributors prioritizing

Continuous batching is "groundwork" (SPEC §12), vague "Now/Next/Later" (ROADMAP), and "Phase 4" (README) — three framings, no single source of truth on what's stable vs. planned.

Evidence: docs/SPEC.md:543–557; docs/ROADMAP.md; README.md:~286.

Fix: one feature-status matrix (stable / experimental / planned) referenced everywhere.

C. Information architecture & docs

How the doc set is organized, navigated, and kept internally consistent.

Friction C1 · Internal design/planning docs are published beside user guides, unlabeled hurts: new users & evaluators

The docs hub auto-lists every docs/*.md alphabetically with no status/maturity marker. Forward-looking design notes (ZINC_RT_DESIGN, GPU_MEMORY_SCALING_PLAN, METAL_PERFORMANCE_PLAN, RDNA4_BATCHED_PREFILL_2X — some labeled "session raw material") sit next to Getting Started. Readers can't tell shipped behavior from aspiration.

Evidence: site/src/lib/docs-loader.ts:67 (sort, no status field); site/src/pages/zinc/docs/index.astro:20–34 (no maturity labels); doc headers e.g. docs/RDNA4_BATCHED_PREFILL_2X.md:1–5.

Fix: add a status:/audience: frontmatter, parse it in the loader, and badge or group "Design / WIP" docs separately.

Friction C2 · Model naming is inconsistent across docs hurts: benchmark reproducers

The same model appears as a CLI id (qwen36-35b-a3b-q4k-xl), a human label (Qwen3.6-35B-A3B Q4_K_XL), and a GGUF filename (...UD-Q4_K_XL.gguf) with no Rosetta. DEVELOPMENT claims "Qwen3, Qwen3.5, and Gemma 4" while the catalog/table shows Qwen 3.5 + 3.6 + Gemma 4 — "Qwen3" / "Qwen3.5" support claims look stale or unverified.

Evidence: docs/GETTING_STARTED.md:26–32, docs/RUNNING_ZINC.md:36,49, docs/BENCHMARKING.md:7, docs/DEVELOPMENT.md:46.

Fix: a model-id ↔ label ↔ GGUF table; reconcile the architecture-support claim with the live catalog.

Hygiene C3 · No prev/next or "related docs" on individual pages hurts: readers following a path

Pages have breadcrumbs, a sidebar, and search, but no footer prev/next and no auto "related." Cross-linking is hand-rolled in prose (Apple trio + SPEC link well; most others dead-end).

Evidence: site/src/layouts/DocLayout.astro:44–68; manual links e.g. docs/APPLE_METAL_REFERENCE.md:7–10.

Fix: sidebar-order prev/next footer; optional related-docs frontmatter.

Polish C4 · Docs hub hides freshness + doesn't separate "Start here" from "Deep docs" visually hurts: trust & orientation

lastmod is loaded but unused on the hub cards, so a reader can't see that some "deep docs" are a month old while others are current. The Start-Here/Deep-Docs split exists on the hub but the sidebar flattens it.

Evidence: site/src/pages/zinc/docs/index.astro (no date on cards); site/src/layouts/DocLayout.astro:44–68.

Fix: show "Updated <month>" on cards; add a sidebar divider/labels.

Polish C5 · Search ranks all doc types equally hurts: search relevance over time

Search boosts a Guide kind, but the loader doesn't assign kinds, so a "performance" query mixes a user guide with a planning doc with no type cue. Low impact today, grows with the design-doc set.

Evidence: site/src/components/DocsSearch.astro:132–138; site/src/lib/docs-loader.ts (no kind).

Fix: tag docs Guide/Reference/Design; rank + badge accordingly.

D. Benchmarks, trust & technical hygiene

Persona: a skeptic deciding whether to believe the numbers — plus site-wide SEO/correctness.

Blocker D1 · Intel target: 4/5 models failing + 5 days stale, still on the dashboard hurts: trust, Intel evaluators

On the Intel tab, only Qwen3-8B succeeds; the other four (incl. the flagship) show ZINC run failed: intel: the execbuf ioctl keeps returning ENOMEM. Intel data is dated 2026-05-19 while RDNA is 2026-05-24. Meanwhile the landing page markets "AMD + Apple" only. A skeptic reads this as a silently-failing node presented as a "supported" comparison.

verified 2026-05-24: site/src/data/zinc-performance.json — intel target, 4 unavailable_reason rows, generated_at 2026-05-19T04:12Z; rendered via coverage cards in BenchmarksDashboard.astro.

Fix: decide Intel's status — either fix the run and refresh, or clearly mark Intel "experimental bring-up, known failures" in the UI, or pull it from the public dashboard. Then make the landing/docs Intel story consistent (see E2).

Friction D2 · /zinc/performance duplicates /zinc/benchmarks (canonical points away) hurts: SEO, navigation clarity

performance.astro renders the same dashboard but sets canonicalPath="/zinc/benchmarks". Two indexable URLs, identical content, link-equity split, and the landing page links to both ("Benchmarks" button → /zinc/benchmarks; perf prose → /zinc/performance historically).

verified: site/src/pages/zinc/performance.astro (5 lines, canonical → benchmarks) vs site/src/pages/zinc/benchmarks.astro.

Fix: pick one canonical URL and 301 the other via _redirects, or give /performance genuinely distinct content (e.g. trends).

Friction D3 · Apple Metal trails llama.cpp materially; the honest caveat is buried hurts: trust for Apple users

Metal overall is ~65% of llama.cpp (flagship decode 33.9 vs 66.3 tok/s); the landing page does say so honestly, but only in a caption under the chart, while "hand-tuned Metal shaders" headlines elsewhere set an expectation of parity. (The honesty is good — the placement undersells the caveat.)

verified: metal flagship ZINC 33.86 vs llama 66.31; site/src/pages/zinc/index.astro caption; ratio in BenchmarksDashboard.astro.

Fix: surface the "Metal is younger / decode trails, prefill leads" framing at the section heading, not just the caption.

Hygiene D4 · Cross-target llama.cpp builds differ; not flagged as a confound hurts: reproducibility skeptics

The baseline llama.cpp version differs per target (RDNA vs Intel vs Metal builds/commits). It's visible in the per-target "Run info" expander but never called out, so a "same baseline?" question resolves to "no" only if you dig.

Evidence: provenance blocks in site/src/data/zinc-performance.json; shown in BenchmarksDashboard.astro run-info.

Fix: note that each target compares against that machine's own current llama.cpp build, with build dates.

Hygiene D5 · Methodology & exact prompts are two clicks / a GitHub hop away hurts: fair-test verification

The methodology card is collapsed per target and the exact prompts/token caps are an inline link into performance_suite.mjs on GitHub. A skeptic can't verify "is this a fair workload?" without leaving the page.

Evidence: BenchmarksDashboard.astro methodology <details> + source link (~lines 1219–1236).

Fix: show the prompt set + run counts inline (or one-click expand) on the page.

Hygiene D6 · Charts encode ZINC vs llama.cpp by color alone hurts: colorblind users (~8% of men)

Bars differ only by orange vs blue; the numeric value is present but the series identity on the bar itself isn't (the legend is separate). Borderline WCAG.

Evidence: BenchmarksDashboard.astro legend/bar markup (~884–945); also the landing-page boards in index.astro.

Fix: label each bar inline ("ZINC", "llama") — already done on the landing boards — and/or add a texture; carry into the dashboard.

Polish D7 · 404 page dead-ends to "the blog" hurts: mistyped /zinc/* URLs

The 404 has a single link: "← Back to blog" (to /). Someone fat-fingering /zinc/benchmark gets no path to ZINC, docs, or benchmarks.

verified: site/src/pages/404.astro (only links /).

Fix: add links to /zinc, /zinc/docs, /zinc/benchmarks.

Polish D8 · llms.txt doesn't point AI crawlers at the benchmarks hurts: AI-answer discoverability

The LLM index lists docs + Zig API but not /zinc/benchmarks or the performance dataset — i.e. the most verifiable "is ZINC faster than llama.cpp?" evidence is omitted for AI crawlers.

Evidence: site/src/pages/llms.txt.ts / llms-full.txt.ts.

Fix: add benchmarks + the JSON dataset URL.

Polish D9 · No benchmark-specific OG image/title for social sharing hurts: link previews

Benchmark/perf links share the generic site OG image; a "128 tok/s on RDNA4" preview card would convert far better in Discord/X.

Evidence: default ogImage in SEOHead.astro; no per-page override in BenchmarksDashboard.astro.

Fix: a benchmark OG image + custom description.

Polish D10 · Raw-data table likely overflows on mobile hurts: phone visitors

The "Raw data" table inside <details> has no mobile-specific reflow/scroll rule; on narrow screens it forces horizontal scrolling. (There is a separate mobile card list, so impact is limited to the raw table.)

Evidence: BenchmarksDashboard.astro .raw-table-details / .bench-table; no narrow-width rule.

Fix: wrap in an overflow-scroll container or hide the raw table on small screens.

Noted but not flagged as a gap: robots.txt blocks GPTBot/ClaudeBot (training crawlers) while allowing ChatGPT-User/Claude-SearchBot (user/search agents). That's a coherent, common policy — worth a conscious confirm, not a defect.

E. Cross-cutting themes

These recur across personas and are worth treating as single initiatives rather than scattered edits.

E1 · "Shipped vs. aspirational" has no consistent signal

TurboQuant (roadmap on the FAQ, but a usable --kv-quant flag in Running ZINC), ZINC_RT (build flag + huge design doc but not the hot path), continuous batching (three different phase labels), and the planning docs in the hub all share one root cause: no maturity vocabulary applied uniformly. A single status taxonomy (Stable / Experimental / Planned) used in frontmatter, the feature table, and the FAQ would resolve A7, B3, B10, C1 at once. touches: A7, B3, B10, C1

E2 · The Intel story contradicts itself three ways

Landing copy = "AMD + Apple" (no Intel). Docs (Getting Started, Hardware Requirements, Intel GPU Reference — 500+ lines) = "Intel Arc supported (experimental)." Benchmarks dashboard = Intel tab with 4/5 runs failing and stale data. Pick one truth and propagate it. If Intel is an experimental target, say so on the landing page and mark the dashboard rows accordingly; if it's not ready, demote it everywhere. touches: D1, and landing/docs consistency

E3 · Information exists, but the "critical path" isn't curated

Across the new-user gaps (A2–A6), the facts are all present — just not in the path a person actually walks. The fix pattern is the same: hoist the must-know item (env var, troubleshooting, model source, fit check) into the first place the user needs it, instead of relying on them reading the whole page. touches: A2, A3, A4, A5, A6, B8

E4 · Numbers are good and honest — keep them that way under a skeptic's eye

The headline RDNA4 win is real (flagship 128.5 tok/s, 1.18× llama.cpp, verified) and losses are reported rather than hidden. The remaining trust work is about surfacing the caveats a skeptic will look for (baseline build differences, failed rows, methodology) rather than hiding or over-claiming. touches: D3, D4, D5

Method & files reviewed
