## Why we're behind, and the lever (the finding that directs this effort)

**Gemma prefill is LAUNCH-BOUND, not compute-bound.** Effort 25 profiled a batched gemma-31B prefill (T=413) at **7–12% GPU util / ~76 W of 575 W — the GPU sits ~90% idle.** So faster GEMM kernels are the WRONG move: Effort 24 proved the batched/tensor-core GEMM is 5.9× isolated but **end-to-end NEUTRAL** (the GPU is idle between kernels, not crunching). The fp16 tensor-core GEMM (`ZINC_BATCHED_TC`, default OFF) compiles and is ~2.2× isolated but is end-to-end neutral for the same reason.

**The lever is the SAME as Effort 25's decode win: CUDA GRAPHS.** Capture the per-layer prefill kernel chain once, `cuGraphLaunch` to replay it, collapsing ~16 launches/layer × 60 layers of per-kernel launch + inter-kernel-bubble latency. The 90% idle is the 4–7× headroom needed to beat llama. Effort 25 already built the capture machinery for decode — extend it to prefill.

## Targets (priority order; the BAR is BEAT llama on the 5090, interleaved-A/B-confirmed)

- **T1 — Gemma prefill via CUDA graphs (PRIMARY).** Extend Effort 25's `cuStreamBeginCapture`/`cuGraphExec` to `prefillBatched` in `forward_cuda_gemma.zig`. **KEY DESIGN:** prefill replays the chain ~ONCE per prompt, so a single whole-prompt graph isn't amortized (capture cost ≈ one replay). **CHUNK** prefill into fixed-size token chunks (e.g. C=128/256), capture the per-C-chunk per-layer chain ONCE, replay it `ceil(T/C)` times — amortizes the instantiate AND cuts the launch chain C-fold. Per-chunk topology is invariant (C fixed); KV cache + causal attention already handle cross-chunk dependencies (each chunk attends prior chunks via the cache). Per-chunk-varying scalars (chunk base position, KV offset) ride device-buffer reads or `cuGraphExecKernelNodeSetParams` so topology stays invariant (same trick decode used). Beat llama: gemma-31B > 381, gemma-26B > 414.
- **T2 — MoE decode gap (gemma-26B 31%, qwen36-35B-A3B 42%).** The router keeps a per-layer host sync and the small-M routed-expert matvecs are occupancy-starved. Levers: (a) **GPU-side expert gather** — drop the last router host round-trip (the e25 doc's MoE-graph blocker), so the MoE step is sync-free; (b) **wider/multi-row expert kernels** for the small-M matvecs (more output rows/block → occupancy); (c) **uniform-quant MoE** — the catalog 35B-A3B has MIXED expert quants (q4k/q5k/q6k) which blocks graph capture; a per-quant-group batched expert kernel (or a requant) unblocks the MoE decode graph. Token-correct gate.
- **T3 — qwen36-35B-A3B MoE prefill (0.94× → beat).** Just +6%. Same chunked-prefill-graph lever as T1 over the MoE prefill chain (batched experts already default-on).
- **T4 (stretch) — dense decode last mile (gemma-31B 82%, qwen35-9B 75%, qwen36-27B 91%).** e25 graphs are size-gated (help 9B, fade at 27B); the lever here is deeper kernel fusion (fewer/fatter launches — the Effort 23 playbook). Lower priority (already close).

I read the current fp16 TC GEMM (`gemm_q4k_tc`, `src/shaders/cuda/kernels.cu:1933`) to scope that rewrite concretely:
- **What it does today:** per K-subblock, dequants the Q4_K weight to **fp16 in shared** — `wv = d*sc*nib - dmin*mn` then `Ws[..]=__float2half(wv)` (`:1972–1974`) — stages the f32/f16 activation, and runs `wmma 16×16×16` fp16×fp16→fp32 (`:1988–1994`). So every weight nibble pays a fp16 dequant + the multiply runs on fp16 TCs (1× Blackwell TC rate).
- **The MMQ change (what llama's MMQ does):** keep the 4-bit nibble as an **int8 operand** (no fp16 dequant of the product), quantize the activation tile to **int8 per-row** (with a per-row f32 scale), and run `mma.sync.aligned.m16n16k16.s8.s8.s32` int8 TCs (**2× the fp16 TC rate on Blackwell, sm_120**), accumulating in s32. Q4_K is **asymmetric** (`w = d·sc·nib − dmin·mn`), so the epilogue must split into two terms:
  - main: `Σ_k A[t,k]·(d·sc·nib)` → factor the per-(row,subblock) `d·sc` out of the s32 int8 accumulator and apply in fp32 epilogue (per 32-wide subblock, since `d·sc` is constant across its 32 nibbles);
  - bias: `−Σ_k A[t,k]·(dmin·mn)` → since `dmin·mn` is constant across each subblock's 32 elems, this is `−(dmin·mn)·(Σ over the 32 A-elems of that subblock)` — i.e. a per-(token,subblock) **A-partial-sum × per-(row,subblock) min**, computable cheaply (reduce A per subblock once, then a small rank-correction). This is exactly the structure llama's `mul_mat_q` epilogue handles.
- **Correctness gate:** int8-quantized A loses precision vs the current fp16-A path → this is NOT byte-identical; it rides the **token-correctness tolerance gate** (`validate_catalog`, same gate `gemm_q4k_tc` already uses, NOT the GEN_IDS byte gate). Must stay catalog 5/5.
- **Why it's the right (and only remaining) lever:** the int8 TCs both **halve the operand bits read** (weight-traffic — and cycle 2 proved the GEMM is weight-traffic-bound) AND **double TC throughput**, attacking both bounds cycle 2 isolated. Multi-cycle kernel build; needs the box to compile/validate/A-B. **Next box-up cycle order:** (1) the cheap, turnkey, ~zero-risk grouped-experts A/B (cycle 3/4, MoE-prefill weight-traffic win, env-gated, no source change) FIRST; (2) then begin the MMQ INT8 dense GEMM above (start with an isolated correctness+microbench of the int8 kernel before wiring it into the prefill chain, mirroring Effort 24's `gemm_q4k_tc` bring-up).

**OFFLINE PRODUCT (new — resolves the hardest open design question in cycle 5's MMQ plan, grounded in re-reading `gemm_q4k_tc` at `kernels.cu:1933–2010`):**
- **The fp16 kernel gets per-(row,subblock) scaling FOR FREE; an int8 mma does NOT.** Today (`:1972–1974`) the weight is dequant'd as `wv = d·sc·nib − dmin·mn` and cast to half *before* the `wmma` — so the per-(row m, subblock) factors `d·sc` and `dmin·mn` are baked into the operand and the tensor core never sees them. - **RESOLUTION — accumulate per-subblock, rescale in shared, fold into an fp32 tile:** structure the K-loop exactly like today (one 32-wide Q4_K subblock per outer step = 2 mma k-steps of m16n16k16). For each subblock: (1) int8 `mma.sync.s8.s8.s32` over the 32 nibbles → a per-(m,t) s32 partial `P = Σ_l nib·qA` (no overflow: 15·127·32 ≈ 6.1e5 ≪ 2³¹); (2) `wmma::store_matrix_sync` that s32 fragment to a shared 16×16 (or 64×64) staging tile so element→(row,token) indices become **known**; (3) in shared, fold `acc_fp32[m,t] += s_A[t]·d[m,sb]·sc[m,sb]·P[m,t] − s_A[t]·dmin[m,sb]·mn[m,sb]·SA[t,sb]`, where `s_A[t]` = per-token int8 quant scale and `SA[t,sb]` = Σ of qA over the subblock's 32 elems (one extra int reduce when staging A). The bias term needs NO mma (rank-1 outer product of per-token `SA·s_A` × per-row `dmin·mn`).
