# ZINC v0.1.0 GitHub Release Plan

Status: draft
Target tag: `v0.1.0`
Release type: GitHub Release with downloadable binaries
Primary goal: publish a trustworthy first release that users can download, verify, and run without building from source.

This is the working plan for the v0.1.0 release. It is intentionally detailed so it can be converted into GitHub issues, a release checklist, or a release workflow. Durable process details can later move into `docs/RELEASING.md`.

## 1. Release Definition

v0.1.0 is the first public binary release of ZINC.

The release is successful when:

- A GitHub Release exists for tag `v0.1.0`.
- The release has downloadable Linux and macOS binary archives.
- Every archive includes the `zinc` executable and all runtime assets needed outside a source checkout.
- Checksums are published for every release archive.
- The release notes clearly state supported platforms, supported model families, known limitations, and validation results.
- A user can download an archive, unpack it, run `zinc --version`, run `zinc --check`, and run a documented smoke prompt.

The release does not need to include:

- Windows binaries.
- Docker images.
- Homebrew packages.
- Linux distro packages.
- Bundled GGUF models.
- Browser or WASM builds.
- Full production server scheduling.
- Full macOS notarization, unless we decide to finish that before tagging.

## 2. Release Audience

The v0.1.0 audience is technical.

Primary users:

- Developers trying ZINC without building from source.
- Contributors who need a stable reference release.
- People validating AMD Vulkan or Apple Metal inference behavior.
- Users comparing ZINC against llama.cpp on known hardware.

Non-goals for this release:

- A polished consumer installer.
- A guarantee that every model in the wild works.
- A support promise for every Vulkan GPU.
- An enterprise deployment story.

The release notes should describe v0.1.0 as an early inference engine release with explicit hardware and model constraints.

## 4. P0 Release Blockers

These must be resolved before tagging.

### 4.1 Runtime Asset Packaging

Release binaries must run outside the repository.

Known risk:

- Metal shaders currently load from source-tree paths such as `src/shaders/metal/{name}.metal`.
- Vulkan releases need compiled SPIR-V shader files available at runtime.

Required implementation:

- Define one stable asset lookup mechanism for release builds.
- Add `ZINC_SHADER_DIR` as an explicit override for debugging and custom layouts.
- Add executable-relative lookup, for example `../share/zinc/shaders`.
- Keep source-tree lookup only as a development fallback.

Preferred v0.1.0 layout:

```text
bin/zinc
share/zinc/shaders/*.spv
share/zinc/shaders/metal/*.metal
```

Alternative:

- Embed shaders in the binary.
- If embedded, document that release archives do not need shader files.

Acceptance criteria:

- `zinc --check` works from an unpacked release archive outside the repository.
- Linux Vulkan runs do not require `src/shaders`.
- macOS Metal runs do not require `src/shaders/metal`.
- Setting `ZINC_SHADER_DIR` overrides the default lookup.

### 4.2 CLI Version Output

Every release binary needs self-identification.

Required command:

```bash
zinc --version
```

Required output fields:

- version, for example `0.1.0`
- git commit hash
- target triple
- optimization mode
- available backend or backend family

Example:

```text
zinc 0.1.0
commit: bfb2a28
target: aarch64-macos
optimize: ReleaseFast
backends: metal
```

Implementation requirements:

- Version should come from the release tag or an explicit build option.
- Commit hash should come from git during build.
- `--version` must exit `0`.
- `--version` must not initialize GPU devices.

Acceptance criteria:

- Users can paste `zinc --version` into bug reports.
- Release notes can instruct users to verify the downloaded binary.

### 4.3 Release Packaging Scripts

The packaging process should not be a one-off shell history.

Required:

- Add a repeatable packaging script or build step.
- Package Linux and macOS archives consistently.
- Generate `VERSION.json` in each archive.
- Generate checksums.

Possible locations:

- `scripts/package_release.sh`
- `tools/package_release.ts`
- `build.zig` release step

Preferred for v0.1.0:

- Keep packaging simple and scriptable.
- Avoid adding runtime dependencies.
- Use shell only if it stays portable and obvious.

Acceptance criteria:

- A fresh checkout at tag `v0.1.0` can produce identical archive layout.
- Packaging fails loudly if runtime assets are missing.

### 4.4 Release Workflow

We need GitHub automation for repeatability.

Required workflow:

```text
.github/workflows/release.yml
```

Required triggers:

- `push` tags matching `v*`
- `workflow_dispatch`

Required jobs:

- Validate tag and source state.
- Run tests.
- Build Linux archive.
- Build macOS archive, either on a hosted Apple Silicon runner or a trusted self-hosted runner.
- Generate checksums.
- Create or update a draft GitHub Release.

For v0.1.0, publishing should remain manual. The workflow should create a draft release so humans can verify artifacts before release.

### 4.5 Hardware Smoke Validation

The release must be tested on actual target hardware.

Required hardware:

- Linux AMD Vulkan machine, preferably the RDNA4 test node.
- Apple Silicon Mac for Metal.

Minimum smoke tests:

- `zinc --version`
- `zinc --check`
- `zinc model list`
- `zinc model list --all`
- one short CLI prompt
- HTTP server health endpoint
- one OpenAI-compatible completion or chat request

Acceptance criteria:

- Validation results are recorded in release notes.
- Known failures are either fixed or documented as limitations.

### 4.6 Release Notes

Release notes must be complete before publishing.

Required sections:

- What is ZINC.
- Downloads.
- Install.
- Quick start.
- Server mode.
- Supported platforms.
- Supported model families.
- Validation matrix.
- Known limitations.
- Checksums.
- Upgrade notes, if applicable.

The release notes must be honest about maturity. Do not imply production readiness beyond what was tested.

## 5. P1 Release Work

These are important but can be scoped depending on time.

### 5.1 Model Catalog Checksums

If release notes recommend managed model IDs, those catalog entries should have complete trust metadata.

Options:

- Fill `sha256` for every recommended model.
- Recommend only models that already have checksums.
- Document checksum gaps clearly.

Preferred:

- Do not recommend a managed model in release notes unless its checksum metadata is complete.

### 5.2 macOS Signing And Notarization

Decision required:

- unsigned binary
- ad-hoc signed binary
- Developer ID signed and notarized binary

Minimum acceptable v0.1.0:

- Document if the macOS binary is not notarized.
- Include a quarantine-removal command if needed:

```bash
xattr -dr com.apple.quarantine ./zinc-v0.1.0-macos-aarch64
```

Preferred after v0.1.0:

- Sign and notarize macOS release artifacts.

### 5.3 Debug Symbols

Decision required:

- ship stripped binaries only
- ship unstripped binaries
- ship stripped binaries plus separate debug symbols

Recommended for v0.1.0:

- Strip normal release binaries.
- Add separate debug symbols later unless crash analysis becomes urgent.

### 5.4 Website Update

After publishing:

- Add a website link to the GitHub release.
- Update any ZINC download or quick-start page.
- Ensure docs do not point users to outdated build-only instructions.

Do this after artifacts are verified, not before.

## 6. Archive Layout

### 6.1 Linux Archive

```text
zinc-v0.1.0-linux-x86_64/
  bin/
    zinc
  share/
    zinc/
      shaders/
        *.spv
  README.md
  LICENSE
  VERSION.json
```

### 6.2 macOS Archive

```text
zinc-v0.1.0-macos-aarch64/
  bin/
    zinc
  share/
    zinc/
      shaders/
        metal/
          *.metal
  README.md
  LICENSE
  VERSION.json
```

If Metal shaders are embedded, omit `share/zinc/shaders/metal`.

### 6.3 VERSION.json

Each archive should include:

```json
{
  "name": "zinc",
  "version": "0.1.0",
  "tag": "v0.1.0",
  "commit": "<git-sha>",
  "target": "<target-triple>",
  "optimize": "ReleaseFast",
  "backend": "<vulkan|metal>",
  "built_at": "<iso-8601-timestamp>"
}
```

Rules:

- `commit` must match the tag commit.
- `version` must match the release tag without the leading `v`.
- `built_at` should be generated by the build system or packaging script.

## 8. Draft Build Commands

These commands are draft examples. The final workflow should wrap them in scripts.

### 8.1 Linux x86_64

Build:

```bash
zig build -Doptimize=ReleaseFast
```

Package:

```bash
version=0.1.0
name="zinc-v${version}-linux-x86_64"

rm -rf "dist/${name}"
mkdir -p "dist/${name}/bin"
mkdir -p "dist/${name}/share/zinc/shaders"

cp zig-out/bin/zinc "dist/${name}/bin/zinc"
cp zig-out/bin/*.spv "dist/${name}/share/zinc/shaders/"
cp README.md LICENSE "dist/${name}/"

tar -C dist -czf "dist/${name}.tar.gz" "${name}"
```

Open item:

- Confirm exact SPIR-V output location from `build.zig`.

### 8.2 macOS Apple Silicon

Build:

```bash
zig build -Doptimize=ReleaseFast
```

Package:

```bash
version=0.1.0
name="zinc-v${version}-macos-aarch64"

rm -rf "dist/${name}"
mkdir -p "dist/${name}/bin"
mkdir -p "dist/${name}/share/zinc/shaders/metal"

cp zig-out/bin/zinc "dist/${name}/bin/zinc"
cp src/shaders/metal/*.metal "dist/${name}/share/zinc/shaders/metal/"
cp README.md LICENSE "dist/${name}/"

tar -C dist -czf "dist/${name}.tar.gz" "${name}"
```

If shaders are embedded, remove the shader-copy step.

### 8.3 Checksums

macOS:

```bash
cd dist
shasum -a 256 *.tar.gz > SHA256SUMS.txt
```

Linux:

```bash
cd dist
sha256sum *.tar.gz > SHA256SUMS.txt
```

The final release should use one combined `SHA256SUMS.txt`.

## 9. GitHub Actions Design

Workflow file:

```text
.github/workflows/release.yml
```

Trigger sketch:

```yaml
on:
  push:
    tags:
      - "v*"
  workflow_dispatch:
    inputs:
      tag:
        description: "Tag to build"
        required: true
        default: "v0.1.0"
```

Recommended jobs:

| Job | Runner | Purpose |
| --- | --- | --- |
| `validate` | Ubuntu | Validate tag, run tests, build once |
| `build-linux-x86_64` | Ubuntu | Build and package Linux Vulkan archive |
| `build-macos-aarch64` | Apple Silicon | Build and package macOS Metal archive |
| `checksums` | Ubuntu | Download artifacts and generate checksums |
| `draft-release` | Ubuntu | Create draft GitHub Release |

Validation job commands:

```bash
zig version
zig build test
zig build -Doptimize=ReleaseFast
```

Draft release command:

```bash
gh release create v0.1.0 \
  --draft \
  --title "ZINC v0.1.0" \
  --notes-file docs/releases/0.1.0-notes.md \
  dist/zinc-v0.1.0-linux-x86_64.tar.gz \
  dist/zinc-v0.1.0-macos-aarch64.tar.gz \
  dist/SHA256SUMS.txt
```

Rules:

- Do not auto-publish v0.1.0.
- Keep the GitHub Release as a draft until manual validation passes.
- Upload logs or build summaries as workflow artifacts if useful.

## 10. Validation Matrix

### 10.1 Build Validation

Before tagging:

```bash
zig build test
zig build -Doptimize=ReleaseFast
npm --prefix site run build
```

The site build is not part of the binary archive, but it catches docs and website regressions before public release.

### 10.2 Linux AMD Validation

Target:

- AMD Radeon AI PRO R9700 or current RDNA4 test node.
- Driver state should match the documented benchmark setup.

Basic checks:

```bash
./bin/zinc --version
./bin/zinc --check
./bin/zinc model list
./bin/zinc model list --all
```

CLI smoke:

```bash
RADV_PERFTEST=coop_matrix ./bin/zinc \
  -m /root/models/Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf \
  --prompt "The capital of France is"
```

Server smoke:

```bash
RADV_PERFTEST=coop_matrix ./bin/zinc \
  -m /root/models/Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf \
  --port 9090
```

Health:

```bash
curl -fsS http://127.0.0.1:9090/health
```

Completion:

```bash
curl -fsS http://127.0.0.1:9090/v1/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"zinc","prompt":"The capital of France is","max_tokens":32}'
```

Record:

- GPU model.
- Driver and Mesa versions.
- Model path or managed model ID.
- Prompt.
- Generated token count.
- Decode tok/s.
- Warnings or errors.

### 10.3 macOS Apple Silicon Validation

Target:

- Apple Silicon Mac.
- Metal backend.

Basic checks:

```bash
./bin/zinc --version
./bin/zinc --check
./bin/zinc model list
```

Managed model smoke:

```bash
./bin/zinc model pull qwen35-9b-q4k-m
./bin/zinc --model-id qwen35-9b-q4k-m --prompt "Hello" --chat
```

If that model is too large for the validation machine, choose a smaller catalog model and document it in the release notes.

Record:

- Mac model.
- Chip generation.
- Unified memory size.
- macOS version.
- Model ID.
- Decode tok/s.
- Whether quarantine removal was required.

## 13. Hotfix And Rollback Plan

GitHub release artifacts can be replaced, but public users may already have downloaded them. Treat published artifacts as immutable unless there is a serious issue.

### Before Public Publishing

If the draft release is wrong:

- replace draft artifacts
- regenerate checksums
- update notes
- rerun validation

If the tag points to the wrong commit and the release has not been published:

```bash
git tag -d v0.1.0
git push origin :refs/tags/v0.1.0
git tag -a v0.1.0 -m "ZINC v0.1.0"
git push origin v0.1.0
```

Only retag before public publishing.

### After Publishing

If v0.1.0 is broken:

- do not silently replace assets without explanation
- mark the release notes with a warning
- publish `v0.1.1` with the fix
- keep `v0.1.0` available unless it contains a security issue

### Security Or Credential Issue

If an artifact or note exposes sensitive data:

- remove affected assets immediately
- rotate exposed credentials if any
- publish a clear advisory or release warning
- publish a corrected `v0.1.1`

## 16. Done Criteria

v0.1.0 is ready to publish when:

- `main` is clean and green.
- Release commit is tagged `v0.1.0`.
- Linux archive is built from the tag.
- macOS archive is built from the tag.
- `SHA256SUMS.txt` verifies both archives.
- Archives unpack and run outside the source checkout.
- `zinc --version` reports `0.1.0` and the correct commit.
- `zinc --check` passes on supported hardware.
- At least one CLI generation smoke test passes on Linux AMD.

- Release notes include validation results.
- Release notes include known limitations.
- GitHub Release is published.
